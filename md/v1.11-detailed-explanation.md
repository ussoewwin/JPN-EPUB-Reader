# v1.11 Detailed Technical Explanation

## Fragment-Range Subheading Grouping & Anchor ID Scoping

---

## 1. The Problem

In commercial EPUBs such as *金田一耕助ファイル20 病院坂の首縊りの家（下）* and *徳川家康（2）*, the table of contents (TOC) and in-book navigation suffered from multiple interlocking failures:

1. **Missing sub-titles**: Small titles like "一", "二", "三" did not appear in the TOC, or only the last section's titles appeared.
2. **Incorrect navigation**: Tapping a sub-title in the TOC always jumped to the **first** occurrence of that title (e.g., every "一" jumped to the first chapter's "一").
3. **Garbled TOC titles**: The `toc.ncx` file declared `encoding="utf-8"` but was actually Shift_JIS, producing mojibake in the TOC.
4. **Flat / duplicated TOC display**: Sub-titles were shown as siblings of their parent instead of nested children, or appeared duplicated.
5. **Missing page breaks**: Sub-titles flowed into the preceding paragraph instead of starting on a new page.

---

## 2. Root Cause Analysis

The fix required resolving **six independent root causes** that compounded each other:

### Root Cause 1 — NCX Encoding Mojibake

Some publishers declare `encoding="utf-8"` in `toc.ncx` while the file is physically encoded in Shift_JIS. The parser trusted the declared encoding blindly, producing replacement characters (`U+FFFD`) and garbage in the TOC.

### Root Cause 2 — Globally-Scoped Synthetic Anchor IDs

Both `EpubParser` and `ContentExtractor` generated synthetic IDs for headings without explicit `id` attributes:

- `__h0`, `__h1`, ... for `<h1>`-`<h6>` tags
- `__ps0`, `__ps1`, ... for `<p class="font-1emNN">` subheadings

These IDs were **only unique within a single XHTML file**. However, `VerticalLayoutEngine` stores all anchor-to-page mappings in a single global `Map<String, Int>` called `anchorPages`. When the same ID (`__h0`) appeared in multiple files, the map only retained the **first** file's page index. Every tap on "一" jumped to the first file's "一".

### Root Cause 3 — Heading Ordinal Mismatch Between Parser and Extractor

`EpubParser` incremented `hOrdinal` for **every** `<h*>` tag found during TOC scanning. `ContentExtractor` only incremented `headingOrdinal` when a tag **lacked** an explicit `id`. This meant:

- File A has `<h3 id="sec1">` then `<h3>` (no id)
  - Parser sees 2 headings → synthetic ID = `__h1`
  - Extractor sees 1 heading → synthetic ID = `__h0`

The synthetic IDs diverged, so the TOC's anchor and the layout engine's anchor pointed to different strings.

### Root Cause 4 — Deduplication Key Collision

`EpubParser.tryAdd()` used `seen.add(plainTitle)` as the deduplication key. In a single file, "一" appears under Chapter 1, Chapter 2, Chapter 3, etc. Because the key was only the text "一", all subsequent occurrences were discarded. Only the first "一" survived in the `cached` list, collapsing distinct subheadings into one.

### Root Cause 5 — PageBreak / Anchor Ordering

`ContentExtractor` emitted `ContentNode.Anchor` **before** `ContentNode.PageBreak`:

```kotlin
nodes.add(ContentNode.Anchor("__h$ord"))
nodes.add(ContentNode.PageBreak)
```

`VerticalLayoutEngine` resolves anchors at the **current page index**. Because the anchor was placed before the page break, it was bound to the page **before** the new page. Tapping the anchor always opened the preceding section.

### Root Cause 6 — No Parent-Range Scoping for Fragmented Files

For files that the NCX splits into many spine entries (`spineUsage >= 2`), the original logic was:

- **Before v1.10**: Skip all subheadings (too slow → no children at all).
- **v1.10 attempt**: Include all subheadings regardless of parent → every subheading in the file attached to the **first** parent TOC entry, causing duplication and wrong grouping.

There was no logic to determine *which* subheadings belonged to *which* parent section within the same file.

## 3. Files Modified & Full Diff

### 3.1 `EpubParser.kt`

#### 3.1.1 `RawSubheading` gains a `level` field

```kotlin
    data class RawSubheading(
        val plainTitle: String,
        val parts: List<TitlePart>,
        val fragId: String,
        /** Heading level. h1=1, h2=2, ..., h6=6. font-1em subheadings = 2. */
        val level: Int,
    )
```

**Why**: The TOC builder (`attachSubheadingChildren`) needs to know each subheading's semantic level to decide whether it qualifies as a child. Previously, only `<h1>`/`<h2>`/`<h3>` were accepted; `<h4>`+ were silently dropped for heavily-fragmented files.

#### 3.1.2 `findSubheadingsInXhtml` now takes `href` and generates prefixed IDs

```kotlin
    private fun findSubheadingsInXhtml(
        xhtml: String,
        chapterDir: String,
        href: String,
    ): List<RawSubheading> {
        val anchorIdPrefix = href
            .replace('/', '_')
            .replace('\\', '_')
            .replace('.', '_')
```

**Why**: `href` (e.g., `text/part0005_split_000.xhtml`) is converted into a safe prefix (`text_part0005_split_000_xhtml`). This prefix is prepended to every synthetic ID so that IDs are unique across the entire book, not just one file.

#### 3.1.3 Synthetic ID generation uses the prefix

For `<h*>` tags:
```kotlin
            val id = if (explicitId.isNotEmpty()) explicitId else "__${anchorIdPrefix}_h$ord"
```

For `<p class="font-1emNN">` / `<span class="font-1emNN">`:
```kotlin
                    val id = if (explicitPId.isNotEmpty()) explicitPId else "__${anchorIdPrefix}_ps$ord"
```

**Why**: `__h0` becomes `__text_part0005_split_000_xhtml_h0`. No two files can collide.

#### 3.1.4 `tryAdd` deduplicates by `plain|id` instead of `plain` only

```kotlin
        fun tryAdd(id: String, parts: List<TitlePart>, level: Int) {
            val plain = buildString {
                for (p in parts) if (p is TitlePart.Text) append(p.text)
            }.trim()
            if (plain.isEmpty() || plain.length > 60) return
            // Within one file, only collapse headings that have the SAME text AND SAME fragId.
            // Different fragId = different physical location = keep as separate entry.
            val key = "$plain|$id"
            if (!seen.add(key)) return
            out.add(RawSubheading(plain, parts, id, level))
        }
```

**Why**: The key `"一|__text_part0005_split_000_xhtml_ps3"` is different from `"一|__text_part0005_split_000_xhtml_ps7"`, so both survive. Only truly duplicate (same text, same fragment) entries are collapsed.

#### 3.1.5 Heading regex extraction records the tag level

```kotlin
            val tagName = m.groupValues[1]
            val level = tagName[1].digitToIntOrNull() ?: 1
            tryAdd(id, parts, level)
```

**Why**: `level` is stored so `attachSubheadingChildren` can use it for filtering and display.

#### 3.1.6 `attachSubheadingChildren` — fragment-range scoping (the core fix)

```kotlin
            val parentFragId = entry.href.substringAfter('#', "")
            // If the next entry points to the same file and carries a fragId,
            // restrict child range to [parentFragId, nextEntrySameFileFragId).
            val nextEntrySameFileFragId = if (
                nextEntry != null && nextStartIdx == startIdx
            ) {
                nextEntry.href.substringAfter('#', "")
            } else ""

            val subs = mutableListOf<TocEntry>()
            var bestFragId = parentFragId
            for (idx in startIdx until endIdx) {
                val sp = spine[idx]
                val cached = scanOrCached(sp)

                val isHeavilyFragmented = (spineUsage[sp.href] ?: 0) >= 2

                val subStart: Int
                val subEnd: Int
                val skipFragId: String
                if (isHeavilyFragmented && idx == startIdx) {
                    // Document-order range: after my parent's fragId,
                    // up to (but not including) the next parent's fragId.
                    val parentIdx = if (parentFragId.isNotEmpty()) {
                        val p = cached.indexOfFirst { it.fragId == parentFragId }
                        if (p < 0) 0 else p + 1
                    } else 0
                    val nextIdx = if (nextEntrySameFileFragId.isNotEmpty()) {
                        val n = cached.indexOfFirst { it.fragId == nextEntrySameFileFragId }
                        if (n < 0) cached.size else n
                    } else cached.size
                    subStart = parentIdx
                    subEnd = nextIdx
                    skipFragId = ""
                } else if (isHeavilyFragmented) {
                    // Other entries for the same file are handled by their own parent.
                    subStart = 0
                    subEnd = 0
                    skipFragId = ""
                } else {
                    subStart = 0
                    subEnd = cached.size
                    skipFragId = if (idx == startIdx) parentFragId else ""
                }

                for (j in subStart until subEnd) {
                    val sub = cached[j]
                    if (skipFragId.isNotEmpty() && sub.fragId == skipFragId) continue
                    if (sub.plainTitle.trim() == entry.title.trim()) continue
                    // ... add to subs ...

                    // Candidate for correcting parent href if it lacks a fragId
                    if (bestFragId.isEmpty() && sub.fragId.isNotEmpty()) {
                        bestFragId = sub.fragId
                    }
                }
            }

            // If the parent href had no fragId, patch it to the first child's fragId
            val correctedEntry = if (bestFragId.isNotEmpty() && parentFragId.isEmpty()) {
                val baseHref = entry.href.substringBefore('#')
                entry.copy(href = "$baseHref#$bestFragId")
            } else entry

            result.add(if (subs.isNotEmpty()) correctedEntry.copy(children = subs) else correctedEntry)
```

**Why**: This is the heart of the fix. For a file split into many spine entries:

1. Find the document-order index of the **current parent's fragment** in the cached subheading list.
2. Find the document-order index of the **next parent's fragment** in the same file.
3. Only subheadings **between** these two indices belong to the current parent.
4. Non-start entries for heavily-fragmented files are skipped entirely (they belong to a different parent).

This turns a flat "all or nothing" approach into precise per-parent scoping.

#### 3.1.7 `decodeWithDetection` — validate declared encoding before trusting it

```kotlin
        fun decodeWithDetection(data: ByteArray): String {
            // BOM check ...

            val declaredEncoding = detectDeclaredEncoding(data)
            if (declaredEncoding != null) {
                try {
                    val charset = Charset.forName(declaredEncoding)
                    val decoded = String(data, charset)
                    if (!looksLikeMojibake(decoded)) {
                        return decoded
                    }
                    // Declared encoding produced mojibake; fall through to detection
                } catch (_: Exception) { }
            }

            // ICU4J detection ...
            for (match in detector.detectAll()) {
                if (match.confidence >= 50) {
                    try {
                        val charset = Charset.forName(match.name)
                        val decoded = String(data, charset)
                        if (!looksLikeMojibake(decoded)) {
                            return decoded
                        }
                    } catch (_: Exception) { }
                }
            }

            return String(data, Charsets.UTF_8)
        }

        private fun looksLikeMojibake(text: String): Boolean {
            var replacementCount = 0
            var controlCount = 0
            for (c in text) {
                when {
                    c == '\uFFFD' -> replacementCount++
                    c.code in 0x0001..0x001F && c !in "\t\n\r" -> controlCount++
                }
            }
            return replacementCount >= 2 ||
                (replacementCount >= 1 && text.length < 200) ||
                controlCount >= 5
        }
```

**Why**: Before, the parser returned `String(data, declaredCharset)` immediately. Now it decodes, then checks for `U+FFFD` (replacement character) and control characters. If mojibake is detected, it falls back to ICU4J's statistical detection, which correctly identifies Shift_JIS even when the XML header lies.

### 3.2 `ContentExtractor.kt`

#### 3.2.1 `extract()` gains `href` parameter and generates prefixed IDs

```kotlin
    fun extract(xhtml: String, chapterDir: String, href: String): List<ContentNode> {
        val nodes = mutableListOf<ContentNode>()
        val anchorIdPrefix = href
            .replace('/', '_')
            .replace('\\', '_')
            .replace('.', '_')
```

**Why**: Must match `EpubParser`'s prefix generation exactly, or TOC anchors and body anchors will never align.

#### 3.2.2 Heading ordinal is incremented for **every** `<h*>` tag

```kotlin
                                    val hOrd = headingOrdinal++
                                    if (elemId.isEmpty()) {
                                        nodes.add(ContentNode.Anchor("__${anchorIdPrefix}_h$hOrd"))
                                    }
                                    nodes.add(ContentNode.PageBreak)
```

**Why**: `headingOrdinal++` moved **before** the `if (elemId.isEmpty())` check. Previously it only incremented for tags *without* explicit IDs. Now it increments for **all** `<h*>` tags, matching `EpubParser`'s `hOrdinal++` behavior and keeping synthetic IDs synchronized.

#### 3.2.3 `PageBreak` is emitted **before** `Anchor`

The old order:
```kotlin
nodes.add(ContentNode.Anchor("__h$headingOrdinal"))
headingOrdinal++
nodes.add(ContentNode.PageBreak)
```

The new order:
```kotlin
val hOrd = headingOrdinal++
if (elemId.isEmpty()) {
    nodes.add(ContentNode.Anchor("__${anchorIdPrefix}_h$hOrd"))
}
nodes.add(ContentNode.PageBreak)
```

**Why**: `VerticalLayoutEngine` maps anchors to `pages.size` at the moment the anchor node is processed. If the anchor comes before `PageBreak`, it maps to the **old** page index. By placing `Anchor` first and `PageBreak` second, the anchor maps to the **new** page (the one started by `PageBreak`).

#### 3.2.4 `PFrame` tracks whether the paragraph contains an `<a>` tag

```kotlin
    private class PFrame(val hasExplicitId: Boolean) {
        var anchored: Boolean = false
        /** True if this <p> contains an <a> tag. Used to skip paragraphs that
         * already contain links from subheading candidacy (matches EpubParser logic). */
        var hasAnchor: Boolean = false
    }
```

And in `<a>` START_TAG processing:
```kotlin
                                "a" -> {
                                    // ...
                                    pStack.lastOrNull()?.hasAnchor = true
                                }
```

**Why**: `EpubParser.findSubheadingsInXhtml` skips paragraphs that contain `<a>` tags (they are usually footnote references, not subheadings). `ContentExtractor` was not skipping them, so `pSpanOrdinal` could drift. The `hasAnchor` flag ensures both sides skip the same paragraphs.

#### 3.2.5 `__ps` synthetic IDs use the prefixed form

```kotlin
                                        nodes.add(ContentNode.Anchor("__${anchorIdPrefix}_ps$ord"))
```

**Why**: Same global-uniqueness requirement as `__h` IDs.

#### 3.2.6 `extractFromBytes` signature updated

```kotlin
        fun extractFromBytes(data: ByteArray, chapterDir: String, href: String): List<ContentNode> {
            val text = EpubParser.decodeWithDetection(data)
            return ContentExtractor().extract(text, chapterDir, href)
        }
```

**Why**: Propagates `href` from the caller.

### 3.4 `VerticalLayoutEngine.kt`

```kotlin
                is ContentNode.PageBreak -> {
                    // ... existing page flush logic ...

                    // When PageBreak is immediately followed by Anchor(s),
                    // pendingAnchorIds must resolve to the NEXT page, not the current one.
                    // Force a new (possibly empty) page so that the anchor maps correctly.
                    if (pendingAnchorIds.isNotEmpty()) {
                        pages.add(Page(currentGlyphs, currentImages))
                        currentGlyphs = mutableListOf()
                        currentImages = mutableListOf()
                        colIndex = 0
                        rowY = firstRowBaseline
                        isFirstGlyphOfColumn = true
                    }
                    pendingParaIndent = false
                }
```

**Why**: Even with `Anchor` placed before `PageBreak` in `ContentExtractor`, `VerticalLayoutEngine` processes the `Anchor` node **before** the `PageBreak` node (they are sequential in the list). When `Anchor` is processed, `pages.size` is still the old value. The anchor is stored in `pendingAnchorIds`. When `PageBreak` is processed, the engine flushes the current page and starts a new one. The added block checks: "if there are pending anchors, flush **another** empty page so that `pages.size` is incremented once more, and the pending anchors resolve to the new page index."

This guarantees that `anchorPages["__prefix_h3"] = pages.size` points to the page **after** the break.

## 4. What Each Change Means

| Change | File | Meaning |
|--------|------|---------|
| `level: Int` on `RawSubheading` | `EpubParser.kt` | Enables the TOC builder to filter and display subheadings by semantic level (h1-h6, font-1em). |
| `href` parameter + `anchorIdPrefix` | `EpubParser.kt`, `ContentExtractor.kt` | Makes synthetic anchor IDs globally unique across the whole book, preventing cross-file collisions in `anchorPages`. |
| `"$plain\|$id"` dedup key | `EpubParser.kt` | Preserves distinct subheadings that share the same text but live at different fragment locations in the same file. |
| `headingOrdinal++` before `if (elemId.isEmpty())` | `ContentExtractor.kt` | Ensures the ordinal counter advances for **every** heading tag, keeping synthetic IDs synchronized with the TOC scanner. |
| `PageBreak` **after** `Anchor` | `ContentExtractor.kt` | Guarantees the anchor is bound to the **new** page, not the page before the break. |
| Empty-page flush on `PageBreak` when `pendingAnchorIds` present | `VerticalLayoutEngine.kt` | Corrects the page index resolution so anchors map to the page that actually contains the heading content. |
| Fragment-range filtering in `attachSubheadingChildren` | `EpubParser.kt` | Restricts child subheadings to the document-order range between the current parent's fragment and the next parent's fragment, preventing duplication and wrong grouping. |
| `looksLikeMojibake()` + fallback | `EpubParser.kt` | Detects when a declared encoding produces garbage and automatically falls back to statistical detection (ICU4J), fixing Shift_JIS files mislabeled as UTF-8. |
| `hasAnchor` on `PFrame` | `ContentExtractor.kt` | Skips paragraphs containing `<a>` tags from subheading candidacy, matching `EpubParser`'s skip logic and preventing ordinal drift. |
| Parent href correction with `bestFragId` | `EpubParser.kt` | If an NCX parent entry lacks a fragment (e.g., points to `part0005.xhtml` without `#id`), patches it to the first child's fragment so navigation lands on the correct section. |

## 5. Summary

v1.11 is not a single bug fix — it is a **coordinated repair of six interacting failures** in the EPUB parsing, content extraction, layout engine, and TOC construction pipeline. Each fix is necessary but insufficient on its own:

- Without **global anchor ID scoping**, fixing the ordinal mismatch would still leave cross-file collisions.
- Without **fragment-range filtering**, allowing all subheadings would duplicate every title under the first parent.
- Without **PageBreak/Anchor reordering** and the **empty-page flush**, anchors would still map to the wrong page even with correct IDs.
- Without **mojibake detection**, the TOC titles would remain unreadable regardless of navigation correctness.

All six fixes must be present simultaneously for the TOC to display correctly, nest hierarchically, and navigate accurately.
